## 1. v2.1 修订目标

v2.1 在 v2.0 基础上增加一个核心能力：**队列状态可持久化**。  
目标是让 `ToDo / Plans / Actions / Attention` 从纯内存态升级为“可观察、可修改、可恢复”的运行态。

### 1.1 关键收益

- 可修改性：可直接编辑快照 JSON 进行调试注入或回放。
- 可移植性：可将运行状态跨机器复制，做问题复现。
- 连续性：重启后可恢复未完成的计划与动作序列。
- 可观测性：队列变化可通过统一快照文件观察，而不是仅依赖日志。

## 3. 队列持久化架构（新增）

### 3.1 数据文件

- 快照目录：`data/queues/`
- 快照文件：`data/queues/runtime_queues.json`

### 3.2 快照结构

```json
{
  "version": "2.1",
  "updated_at": 0.0,
  "reason": "autosave|heartbeat_tick|agent_stop|manual",
  "queues": {
    "todo": [],
    "plans": [],
    "actions": [],
    "attention": null
  }
}
```

### 3.3 生命周期

1. **自动保存**：队列 push/pop/clear/replace 与 attention 变更时触发（可配置）。
2. **心跳保存**：每个 `tick` 末尾强制保存一次，保证 Attention 进度与动作队列一致。
3. **停机保存**：`agent.stop()` 时保存最终状态。
4. **启动恢复**：`agent.start()` 时先尝试从快照恢复，再决定是否注入 demo ToDo。

### 3.4 关键约束

- `Urgency` 与 `AttentionState` 以字符串持久化（enum.value），便于人工编辑。
- 恢复失败时降级为干净启动，不阻塞引擎。
- 恢复成功后不会重复注入 demo ToDo（仅在运行态为空时注入）。

## 4. 配置项（v2.1 新增）

| 配置项                    | 默认值                            | 说明                     |
| ------------------------- | --------------------------------- | ------------------------ |
| `QUEUES_AUTOSAVE`         | `true`                            | 队列变化时是否自动写快照 |
| `QUEUES_RESTORE_ON_START` | `true`                            | 启动时是否从快照恢复     |
| `QUEUES_SNAPSHOT_FILE`    | `data/queues/runtime_queues.json` | 队列快照文件路径         |

## 5. 运行与观测建议

### 5.1 最小观测面

- `logs/polaris.log`：看行为与阶段流转。
- `data/queues/runtime_queues.json`：看当前运行状态与可恢复上下文。

### 5.2 推荐调试流程

1. 启动内核跑到中间状态。
2. 停机后编辑 `runtime_queues.json`（例如插入新 Todo 或修改 Plan priority）。
3. 重启并验证恢复后的执行轨迹是否符合预期。

## 7. 风险与后续改进

- 高频 autosave 会带来 IO 开销；后续可改为节流保存（例如每 N 秒合并写入）。
- JSON 人工可编辑带来误配置风险；后续可增加 schema 校验和坏档隔离。
- 当前快照仅覆盖队列层；后续可选扩展到 `BotState` 的部分字段快照。
