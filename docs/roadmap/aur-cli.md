# AUR CLI 路线图

AuroraBot 后续希望把当前的应用测试入口 `app.py` 演进为统一的应用工具链，对外统一使用 `aur` 这个名字。

## 目标

`aur` 主要承担两类职责：

- 开发期工具：扫描应用、按配置加载、调用命令、注入事件、执行 tick
- 分发期工具：打包 `.aur`、安装应用、卸载应用、初始化配置

## 命名约定

- CLI 工具名使用 `aur`
- 包格式后缀使用 `.aur`
- 文档、帮助文案和脚本入口逐步统一到 `aur`

## 它和现状是什么关系

当前项目已经有一些基础能力可以直接收束到 `aur`：

- 目录发现与实例化应用
- 应用级配置管理
- 独立测试入口 `app.py`
- 每个应用的 `README.md` 与 `config.example.json`

因此更合理的方向是“演进与收口”，而不是推翻重写。

## 设计边界

`aur` 只处理应用层与控制平面的事情，不直接承担内核推理职责。

### 它应该负责

- 发现有哪些可用应用
- 查看应用是否启用
- 直接调用应用命令
- 注入测试事件
- 执行测试 tick
- 打包、安装、卸载应用

### 它不应该负责

- Brain 认知推理
- Kernel 调度策略
- 系统级环境变量管理

## 推荐的命令结构

| 命令              | 作用                      |
| ----------------- | ------------------------- |
| `aur list`        | 查看已发现应用及启用状态  |
| `aur run`         | 按配置启动启用的应用      |
| `aur command`     | 直接调用某个应用命令      |
| `aur event`       | 注入测试事件              |
| `aur tick`        | 手动执行若干次 tick       |
| `aur config init` | 初始化 `apps/config.yaml` |
| `aur pack`        | 打包应用为 `.aur`         |
| `aur install`     | 安装 `.aur` 包            |
| `aur uninstall`   | 卸载应用                  |

## `.aur` 包格式建议

第一版可以直接基于 zip，后缀改成 `.aur`。

一个最小可用的应用包至少应包含：

- `manifest.yaml`
- `__init__.py`
- `runtime.py`
- `README.md`
- `config.example.json`
- `aur.yaml`

## 配置哲学

应用配置建议继续分两层：

| 配置位置                         | 作用                             |
| -------------------------------- | -------------------------------- |
| `apps/config.yaml`               | 平台视角的应用启用状态与启动参数 |
| `app-data/<package>/config.json` | 应用私有运行配置与持久化参数     |

这样可以保证：

- 平台只关心“如何启动”
- 应用自己关心“如何解释自己的配置”

## 分阶段落地建议

### 第一阶段

- 把 `app.py` 改造成子命令结构
- 对外帮助文案统一使用 `aur`

### 第二阶段

- 确定 `aur.yaml`
- 明确 `.aur` 的目录结构与校验规则

### 第三阶段

- 完成 `pack / install / uninstall`
- 增加配置自动补全

### 第四阶段

- 增加 `--force`
- 增加 `--purge-data`
- 增加版本冲突处理
- 增加校验和或签名机制

## 当前结论

把应用工具链统一命名为 `aur` 是合理的，因为它既适合承载开发期工具，也适合承载后续的打包与分发能力。
